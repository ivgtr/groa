# リスク・前提条件

## 前提条件

- Node.js 20+ がインストールされていること
- pnpm がグローバルにインストールされていること
- `api` バックエンド使用時は Anthropic APIキーが取得済みであること
- `claude-code` バックエンド使用時は Claude Code CLI がインストール済みで認証されていること
- ユーザーのツイートデータが spec.md §7.3 のJSON形式で準備されていること
- 入力ツイートデータの言語は日本語を主対象としている（kuromoji.jsの形態素解析が日本語前提）
- Web版使用時はES2022+対応のモダンブラウザを使用すること

## 不明点【要確認】

- [ ] kuromoji.js の辞書をWeb版でどこからホストするか（CDN / 同一ドメイン / 初回DL）の配信戦略
- [ ] Transformers.js の ONNX モデルの配信・キャッシュ戦略（Web版での初回118MBダウンロードのUX）
- [ ] Batch API の応答時間が30分を超えるケースの実際の頻度と、ユーザーへの待機UXの適切さ
- [ ] Web版の IndexedDB 容量上限（ブラウザ依存）が EmbeddingIndex（約25MB JSON）+ 中間結果の格納に十分か
- [ ] Twitter/X公式エクスポート形式のフィールドマッピングの具体的な変換仕様（READMEに記載する案内の詳細度）
- [ ] `anthropic-dangerous-direct-browser-access` ヘッダ使用時のセキュリティリスクをユーザーにどの程度詳しく説明するか
- [ ] spec.md §9.4 の倫理ガイドラインにおける利用規約の具体的な文面（法務レビューの要否）
- [ ] コスト見積の精度（実際のトークン使用量はプロンプト設計に依存するため、大幅に乖離する可能性）
- [ ] 日本語以外の言語のツイートが混在した場合のkuromoji.jsの挙動と品質への影響
- [ ] `groa.json` のスキーマバージョニング（将来の設定項目追加時のマイグレーション方針）

## リスク

- **kuromoji.js のWeb Worker対応**: kuromoji.jsは元々Node.js向けで、Web Worker内での辞書読み込みに追加の対応（パスの解決、fetchでの取得等）が必要になる可能性がある。見積もりが膨らみやすい
- **Transformers.js のブラウザ互換性**: ONNX Runtime WebAssembly のブラウザ互換性に問題が生じる可能性。特にSafariでのWebAssembly SIMD対応
- **Batch API のレイテンシ**: Anthropic Batch APIの応答時間は保証されておらず、高負荷時に60分のタイムアウトに到達するリスク
- **Embedding IndexのJSONシリアライズサイズ**: 50,000件の場合、Float32Arrayの JSON表現が非常に大きくなる（約150MB）。メモリ消費とI/Oがボトルネックになる可能性
- **LLMレスポンスの品質安定性**: Haikuの分類精度、Sonnetの分析・評価の安定性、Opusのペルソナ合成品質が、モデルバージョンアップで変動するリスク
- **Prompt Cachingの効果**: キャッシュヒット率はリクエスト間隔に依存し、バッチ的に複数件を連続生成しない場合はコスト削減効果が限定的
- **Web版全体のパフォーマンス**: kuromoji.js辞書展開（約50MB）+ Transformers.jsモデル（約118MB）をブラウザで扱うため、低スペック端末でのメモリ不足リスク
- **テスト用合成データセットの品質**: 合成データが実データの特性を十分にカバーしない場合、テストが実運用時の問題を検出できない
