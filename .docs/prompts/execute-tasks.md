# タスク自動実行プロンプト

`.docs/tasks/` 配下のタスクを依存順にすべて実行し、プロジェクトを完成させてください。

---

## 前提

- タスク定義は `.docs/tasks/items/` に1ファイル1タスクで格納されている
- 各タスクファイルにはステータス・優先度・完了条件・依存関係が記載されている
- `.docs/tasks/README.md` にタスク一覧と進捗サマリが存在する
- 要件書は `.docs/spec.md` にある（実装の判断に迷った場合に参照すること）

---

## ループ手順

以下の Step 1〜8 を、終了条件を満たすまで繰り返してください。

### Step 1: 現状把握

`.docs/tasks/README.md` のタスク一覧を読み、全タスクの現在のステータスを確認する。

### Step 2: 次のタスクを選択

以下の条件をすべて満たすタスクの中から、**優先度が最も高く、IDが最も若い**ものを1つ選ぶ。

選択条件:
1. ステータスが `Todo` である
2. 先行タスク（依存関係の「先行」欄）がすべて `Done` である
3. 優先度の順序: Must > Should > Could

**該当タスクが1つもない場合 → Step 8（終了判定）へ進む。**

### Step 3: タスク実行

選択したタスクファイル（`.docs/tasks/items/T[ID].md`）を読み込み、以下を順に行う。

1. ステータスを `In Progress` に更新して保存する
2. 「目的」「完了条件」「実装メモ」を把握する
3. 必要に応じて `.docs/spec.md` の該当箇所を参照する
4. 実装を行う

**実装中の制約:**
- このタスクのスコープを超える変更をしないこと
- 既存のテスト・型チェック・lintを壊さないこと
- 判断に迷う設計上の選択は、タスクファイルの実装メモに決定理由を追記すること
- 実行不能な不明点が発生した場合 → ステータスを `Blocked` に変更し、理由を実装メモに記録して **Step 7 へ**

### Step 4: 完了条件の検証

タスクファイルに記載された完了条件を **1つずつ** 機械的に検証する。

検証の実行例:
| 完了条件の記述 | 検証コマンド |
|--------------|------------|
| 型エラーが0件 | `npx tsc --noEmit` 等 |
| テストが通る | `npm test` / `bun test` 等 |
| lintエラーが0件 | `npx eslint .` / `npx biome check .` 等 |
| ビルドが成功する | `npm run build` 等 |

**全条件を満たした場合 → Step 5 へ。**
**満たせない条件がある場合 → 修正を試みる。同一条件に対して3回修正を試行しても解決しない場合、ステータスを `Blocked` に変更し、失敗条件と原因を実装メモに記録して Step 7 へ。**

### Step 5: セルフレビュー

完了条件を満たした実装に対して、以下の観点でセルフレビューを行う。
**指摘が0件になるまで、修正 → 再レビューを繰り返すこと。**

#### レビュー観点

1. **要件との整合性**
   - タスクの「目的」に記載された意図を満たしているか
   - `.docs/spec.md` の該当要件から逸脱していないか

2. **コード品質**
   - 不要なコード・コメントアウト・デバッグ用コードが残っていないか
   - 命名は意図を正確に表しているか
   - 重複ロジックがないか
   - エラーハンドリングは適切か

3. **副作用の確認**
   - このタスクで変更したファイル以外に影響が出ていないか
   - 既存テストがすべて通るか（Step 4 の検証コマンドを再実行）

4. **タスクスコープの逸脱**
   - 完了条件に書かれていない変更を含んでいないか
   - 含んでいる場合、その変更を元に戻すこと

#### レビューの進め方

1. 上記4観点それぞれについて、指摘事項をリストアップする
2. 指摘が **1件以上** ある場合:
   - 修正を行う
   - Step 4（完了条件の検証）を再実行する
   - Step 5 のレビューを再度行う
3. 指摘が **0件** になった場合 → Step 6 へ進む

**レビュー→修正のサイクルが3周を超えた場合:**
残っている指摘事項をタスクファイルの実装メモに `## レビュー残課題` として記録し、Step 6 へ進む。（無限ループを防止する）

### Step 6: ステータス更新（完了）

以下の2ファイルを更新する:

**タスクファイル（`.docs/tasks/items/T[ID].md`）:**
- ステータスを `Done` に変更
- 完了条件のチェックボックスをすべて `[x]` にする

**インデックス（`.docs/tasks/README.md`）:**
- タスク一覧の該当行のステータスを `Done` に変更
- 進捗サマリの完了数を +1 する

### Step 7: ループ継続

**Step 1 に戻る。**

### Step 8: 終了判定

以下のいずれかに該当する場合、ループを終了する。

| 終了パターン | 条件 |
|------------|------|
| 正常終了 | すべての Must + Should タスクが `Done` |
| 部分終了 | `Todo` のタスクが残っているが、すべて依存未解決または `Blocked` |
| Could 残存 | Must + Should がすべて `Done` で、残りが `Could` のみ |

終了時、`.docs/tasks/README.md` の末尾に以下を追記する:

```markdown
## 実行サマリ（YYYY-MM-DD HH:MM）
- 完了: N タスク
- ブロック: N タスク（理由は各タスクファイルを参照）
- 未着手: N タスク
```

---

## 制約事項

- **1ループ = 1タスク**: 複数タスクを同時に進めない。必ず1つ完了またはBlockedにしてから次へ進む
- **ステータス更新は即時**: 着手前に `In Progress`、完了/失敗後に `Done` / `Blocked` を書き込む
- **スコープ厳守**: タスクファイルに記載されていない機能追加・リファクタリング・最適化をしない
- **Blocked は飛ばす**: Blocked タスクの解消は試みず、次の実行可能なタスクへ進む
- **Could は自動実行しない**: Must + Should が完了した時点で終了する
- **レビューは3周まで**: セルフレビューの修正サイクルが3周を超えたら残課題を記録して完了扱いとする
